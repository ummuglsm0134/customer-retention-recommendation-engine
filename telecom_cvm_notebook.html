3 · Customer Segmentation

Before any recommendations are generated, each customer is assigned to a value segment (A–E) based on their monthly margin contribution. Segmentation uses quantile-based binning on the most recent month-end record per customer, after filtering out employee accounts and anomalous margin values.

  • 5 quantile buckets (A = highest value, E = lowest)
  • Customers who switch segments over time are tracked separately — these rank transitions are the signal for pattern-based recommendations
  • Employee IDs are excluded to prevent internal accounts from skewing the model

4 · Pattern-Based Engine (Collaborative Filtering)

The pattern-based engine uses item-item collaborative filtering (cosine similarity on a customer–package purchase matrix) to identify which packages tend to be purchased together, then applies user-item scoring to rank those packages for each individual customer based on their value segment.

Two-stage pipeline

  • Stage 1 — Item-Item Similarity: For each value segment, build a customer × package binary matrix and compute pairwise cosine similarities. Top-K similar packages are stored as a knowledge base.
  • Stage 2 — User-Item Scoring: For each customer in segments B–E (segment A customers already have the best packages), score candidate packages using a weighted similarity function and return the top 6.

5 · Rule-Based Engine

The rule-based engine applies explicit business logic to filter and rank package recommendations based on a customer's current services, technology tier (Copper / Coaxial / Fiber), and gross profit margin (GPM) thresholds. Five rule types run in parallel per customer via Ray remote tasks.

  • Package rules — don't offer packages containing services the customer already has
  • Standalone rules — offer à la carte options to single-product customers
  • Pick-option rules — upsell to higher tier packages matching current usage
  • Rooftop rules — location-based eligibility filtering
  • Campaign rules — seasonal / promotional package injection

7 · Key Design Decisions

Why cosine similarity for package recommendations?

Package adoption data is sparse and binary (customer has / doesn't have a package). Cosine similarity on binary vectors handles this well — it measures the angle between purchase patterns rather than magnitude, making it robust to varying customer tenure lengths.

Why Ray for parallelism?

The rule engine must run 5 independent rule checks for each of ~8,000+ customers. Sequential execution would take 30–40 minutes. Ray's async remote task model brings this down to under 3 minutes by distributing 300-customer batches across CPU cores.

Why keep two engines separate?

🧠 Pattern Engine

Learns from behavioral data — good at surfacing non-obvious package affinities. Adapts as customer mix changes over time.

📋 Rule Engine

Enforces business constraints — ensures recommendations are technically eligible, margin-positive, and campaign-aware. Prevents ML from suggesting invalid combos.

9 · Summary & Learnings

✅ What was built

A production recommendation engine serving thousands of telecom customers daily, combining ML-based collaborative filtering with business-rule validation — deployed end-to-end from Python data pipeline to C# real-time API.

Tech stack

Python 3.10 pandas / numpy scipy (cosine sim) Ray (distributed) SQLAlchemy SQL Server C# / SignalR API

Key takeaways

  • Segment-aware recommendation (A–E tiers) significantly reduces irrelevant suggestions
  • Hybrid pattern + rule approach prevents ML from recommending technically ineligible packages
  • Ray parallelism reduced end-to-end runtime by ~10× vs. sequential processing
  • Delta run logic prevents re-processing customers with no new activity
